Here is a detailed breakdown of the deep technical issue occurring when `hv_guest` attempts to use PowerShell Direct against a Virtual Machine that does not have an operating system installed.

### The Underlying Technology: PowerShell Direct (VMBus)
Standard PowerShell remoting (WinRM) connects over TCP/IP (usually port 5985/5986).
PowerShell Direct, however, connects over the **VMBus**. The VMBus is a high-speed memory-sharing conduit between the Hyper-V hypervisor (the Host) and the Virtual Machine (the Guest).

When you run `Invoke-Command -VMName ...`, the Hyper-V host reaches out over the VMBus and attempts to negotiate a connection with the **Hyper-V Integration Services (specifically the PowerShell Direct daemon)** running inside the Guest OS.

### The Test Environment Setup
In our CI environment (`tests.yml`), we create a "dummy" test VM:
```yaml
    - name: Setup - Create VM
      microsoft.hyperv.hv_vm:
        name: "{{ test_vm_direct }}"
        state: present
        generation: 2
```
This creates a blank VM. It has a CPU, RAM, and a synthetic network adapter, but **it has no virtual hard disk and no Operating System installed.**

### The Execution Flow & Failure
When the test playbook executes the `action: run` command:
1.  The `hv_guest.ps1` module validates the inputs.
2.  It checks the VM state. The VM is `Running`, so it proceeds.
3.  It runs `Invoke-Command -VMName "AnsibleDirectVM_123" ...`
4.  The Hyper-V Host attempts to send the command over the VMBus to the guest.
5.  **The Failure Point:** Because the Guest VM has no OS, there is no Integration Service daemon listening on the other side of the VMBus.

### The Exception Geometry
When the VMBus connection times out, PowerShell throws an extremely complex, deeply nested exception object. It is usually a `PSRemotingTransportException` or `PSRemotingDataStructureException`.

This is not a simple string. It is a massive C# object containing connection metadata, transport traces, and remote runspace tracking information.
